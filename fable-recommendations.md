# run-mcp — Recommendations & Prioritized Roadmap

*Phase-3 synthesis: my independent findings (`fable-analysis.md`) × your ideas (`.invisible/ideas.md`) × the two research reports (in-scope sections). Reactions first, then the tiered roadmap.*

---

## 1. Reactions to your ideas

**Idea 1 — "SDK for code? something you could import in a JS app — is there value over what the TypeScript SDK provides?"**
Yes, there is value, and the reason is precise: the official SDK — including the v2.0.0-alpha rewrite in your workspace clone — still has **no message-level middleware hook on the client path**. (The v2 monorepo's `middleware` package is HTTP-framework glue — Express/Hono/Fastify adapters — not message interception; the SEP-1763 gap persists.) So `wrapClient(client, [plugins])` would offer something the SDK genuinely doesn't. **But your locked decision #4 (defer until a real external consumer exists) remains correct**, with one sharpened caveat: the v2 SDK is a breaking rewrite mid-flight (they ship a codemod for the migration). Publishing a library against v1.x today means porting it in months. Deferring is not just discipline — it's good timing. What you should do *now* is cheap: treat `InterceptorPlugin` as a public API in waiting (freeze or version its signature, document it), because third parties will write plugins before you declare it stable. Verdict: **right instinct, right deferral, one prep task.**

**Idea 2 — "Something that helps with context bloat because of how harnesses implement MCP — out of scope?"**
This was the seed of the compressing proxy, and the specific phrasing contains the strategic insight that makes it *in* scope: the bloat is caused by **client/harness behavior** (eager schema injection) that neither you nor server authors control — so the only universal fix lives in the pipe between them, which is exactly where run-mcp already sits. Verdict: **in scope, correctly built, and the analysis (§2) says the next move is finishing it properly** (cache, fidelity, result-side) rather than questioning it.

## 2. Reactions to the research

### Where it reinforces my independent view
- **The Tools-Tax framing and magnitude** (10k–60k tokens/turn; GitHub's 94 tools ≈ 17.6k tokens; catalogs that can't physically fit) matches what the code addresses. The DCL three-tier hierarchy in the critiques doc is *literally* the B2 surface you shipped (`list_servers` → `find_tools` → schema → invoke). The research and the implementation agree.
- **The SEP-1763 interceptor-gap table** is the strongest single validation of the plugin framework as the strategic asset — seven enumerated gaps, and run-mcp's hooks already cover input-adjacent validation, output transformation, cross-cutting logging, and (embryonically) policy. My Phase-2 conclusion ("the programmable middle of the pipe") is the same conclusion by another route.
- **The binary-payload / composability critique** ("file content forced through the context window multiple times; no tool-to-tool data passing without the LLM as intermediary") is precisely the pain my #1 unbuilt feature targets: **result spill-to-disk + `read_result` handle**. The research strengthens the case and extends it: handles shouldn't just be readable, they should be *passable* — `invoke_tool` accepting a handle as an argument value (proxy-side dereference) gives you tool→tool data flow without the model copying payloads. That upgrade came directly from reading the research; I'd fold it into the feature.
- **Stdio stream-hygiene emphasis** (stdout pollution kills sessions) validates existing behavior — TargetManager pipes stderr and the SDK transport handles framing — and suggests a cheap new *validator check*: "server writes non-JSON to stdout" as an explicit FAIL with the offending bytes shown. Nice fit for the dev-tool identity.

### Where it contradicts me — and where I push back
- **"Semantic embeddings are strictly necessary" (the 8.1pp TF-IDF-downgrade claim).** This is the research's headline argument for shipping a local MiniLM (transformers.js, ~22MB, q8). I don't buy it as stated, for three reasons. (1) The claim traces to a single benchmark ("Tool Attention", simulated 120-tool environment) with suspiciously precise ablations and no reproduction I'm aware of — treat as directional, not gospel. (2) The strongest counter-evidence ships in production: Anthropic's own tool-search feature defaults to regex/BM25 with embeddings as the *optional* upgrade — if lexical ranking cost 8 points of task success at that scale, that default would be indefensible. (3) BM25 ≠ TF-IDF; saturation + length normalization + your name-boosting close part of the gap the ablation measured. **Keep BM25; make the embeddings decision empirical**: the eval harness (roadmap §Strategic Bets) should include a tool-retrieval benchmark over real catalogs — if BM25's hit-rate@k actually sags there, Stage C (opt-in embeddings) is already planned and the `ranking.ts` module boundary makes it a drop-in. Decide from your own data, not one paper.
- **The bigger flaw: Tool-Attention-style routing doesn't fit an MCP proxy's vantage point.** ISO scoring computes similarity between **the user's query** and tool descriptions *before the model runs*. A middleware inside an orchestration framework (LangGraph pre-model hooks) sees the user's prompt; **an MCP proxy does not** — it sees tool calls, not the conversation. The blueprint hand-waves this ("upon intercepting an agent's prompt or inferring its reasoning step") but there is no such interception point on the MCP wire. This is not a nitpick; it means the research's marquee architecture is structurally unavailable to run-mcp, and the design you actually shipped — where the *model* formulates the query and calls `find_tools` — is the correct adaptation to the proxy's position, not a lesser fallback. The same constraint kills "stateful gating on user authorization" at the proxy layer in its full form (though per-tool allow/deny policy — which *is* visible to a proxy — survives as the policy-engine feature).
- **The "always-on inflation guard, 75–89% output reduction" figures** (OmniRoute/Caveman cites): your `--compress-output` implements the guard correctly, but real-world lossless-minify savings are nowhere near 75–89% on typical payloads; those numbers describe aggressive summarization/truncation regimes. Don't chase them with more whitespace tricks — structural moves (spill-to-disk, schema-aware pruning, response diffing) are where order-of-magnitude reductions actually live.
- **General calibration note:** both reports show signs of synthesis-with-confidence (very precise uncited percentages, a formalized "Total Attention Energy" concept I can't find independent grounding for, image placeholders). The *directional* claims are sound and useful; I'd treat every specific number as marketing-grade, not engineering-grade.

### What the research adds that I'd adopt
- **SKILL.md progressive disclosure as a proxy feature** (context-bloat doc §Skills). Serving local skill directories as lazily-loaded MCP capabilities (frontmatter summary → full body on activation → `run_skill_script`/`read_skill_resource` on demand) is a genuinely good fit: it reuses the entire DCL surface, costs no heavy deps, and positions run-mcp at the intersection of the two capability-packaging standards (MCP tools and Agent Skills). Promoted to a Strategic Bet below.
- **Prompt-cache stability as a first-class design goal** (compressor doc): the wrapper surface being *immutable across turns* is a large hidden win (cache hits). Two implications for run-mcp: fix the stale-catalog bug in a cache-*friendly* way (the `get_tool_schema` description should change **only** when the catalog truly changes, never per-request), and treat "stable tool surface" as a constraint on future features (don't dynamically mutate wrapper descriptions per turn).
- **Code Mode's fixed-ceiling argument** (Cloudflare: 2,500 endpoints behind `search` + `execute` ≈ 1k tokens): the strongest articulation of why the compression staircase ends at *composition*. It confirms Code Mode as the right long-term bet and confirms your call to defer the sandbox — but the fixed-ceiling framing is worth adopting *now* in how you document the proxy's levels.
- **What the research doesn't cover at all** — and my analysis flags as urgent: the SDK v2 **era split** (`initialize` removal, `_meta` envelopes, tasks-as-extension), **OAuth on outbound connections**, and **backend lifecycle** (reconnect/lazy-spawn). The research is a context-compression document; the proxy's production-readiness gaps are outside its frame. Don't let its thoroughness on compression imply the compression work is the critical path — fidelity and lifecycle are (analysis §2.4).

## 3. Prioritized roadmap

### Tier 1 — Fix Now *(correctness, leaks, and the proxy's credibility gaps; ~1–2 weeks total)*

1. **Interceptor timeout-timer leak** — `clearTimeout` when the race settles (`interceptor.ts:_timeout`). One-liner-ish, affects every mode.
2. **Tool-list cache in the proxy** — per-backend cache, invalidated by `tools/list_changed` (TTL fallback), parallel fan-out in `find_tools`/overview, cursor-loop for paginated backends, and regenerate the embedded catalog description from the same cache (fixes the stale-catalog/fresh-handler split; do it cache-*stably* per §2 above). This single item removes the proxy's worst latency, correctness, and scale problems at once.
3. **Proxy sampling/elicitation forwarding** — wire the `sampling_request`/`elicitation_request` listeners in `proxy.ts` the same way `server.ts` does. Without it, any backend that uses sampling hangs 5 minutes behind the proxy. ~20 lines.
4. **Backend reconnect + honest failure surfacing** — `enableAutoReconnect()` on pooled targets; `list_servers` shows connection state; `invoke_tool` on a dead backend says "backend X is down," not "tool not found."
5. **Cassette write batching** — stop rewriting the whole file synchronously per record (debounced flush / JSONL append). Prereq for any "record a long session" feature.
6. **`TargetManager._instances` removal on `close()`** + cap what `_history` retains (sizes/hashes beyond a small window, not full multi-MB results).
7. **Namespace hardening** — collapse `_` runs in `normalizeServerName` so no prefix can contain `__`; add proxy-of-proxy test.
8. **`firstSentence` robustness** ("e.g.", "v1.2", URLs) + a catalog-size guard that auto-escalates compression level with a logged notice.
9. **Lazy-load the validator** so `ajv` compilation stops taxing every CLI startup; drop the redundant pre-call `tools/list` in `call_mcp_primitive` (reuse cache).
10. **Test hygiene sweep** — kill the real example.com egress; consolidate `mockTarget`/tmp-dir/skip-guard helpers into `helpers.ts`; add the missing auto-reconnect *success-path* test and a dead-backend proxy test; replace sleep-based syncs with event waits where cheap.
11. **Graceful signal shutdown** — bounded `Promise.allSettled` grace before `process.exit` so tree-kills actually land.

### Tier 2 — Build Next *(the strategy, made real; roughly in order)*

1. **Full-fidelity passthrough substrate.** Restructure the proxy so *everything* forwards by default — resources, prompts, notifications (incl. `list_changed` propagation downstream), progress, cancellation, `structuredContent`, user `_meta`, pagination — and compression/DCL becomes a *transformation layered on top*. Build it with a version/codec seam (the SDK's era split is coming: `initialize` removal, reserved-`_meta` lift-and-restamp — never blindly forward connection-scoped `_meta` keys). This is the item that makes "dev-tool + production-proxy" one product instead of two; per the analysis it's prerequisite to the word "production."
2. **Result spill-to-disk + `read_result` + pass-by-handle.** Full result written to `outDir`; agent gets head + stats + handle; `read_result(handle, range/path)` navigates; handles accepted as `invoke_tool` argument values for tool→tool data flow without model round-trips. Highest-leverage genuinely-novel feature in the pipeline.
3. **Unify B1/B2 proxy surfaces** — single backend = multiplexer with one entry (prefixing off). One code path; `find_tools` available for large single servers, which are arguably the primary compression case.
4. **OAuth client support for remote targets** — thread the SDK's `authProvider` (PKCE, RFC 9728 discovery, RFC 8707) through `TargetManager` for REPL/headless/proxy alike. Unblocks the entire remote-server segment for every mode at once.
5. **Owned config file (`run-mcp.config.json`)** — backends, compression level, plugins, per-server policy, outDir; the CLI flags become overrides. A production proxy is configured, not flagged. (You already parse `mcpServers` shapes; this is formalizing what exists.)
6. **Session stats (`--stats` / `stats` REPL command / end-of-session line)** — tokens saved, calls, latency percentiles per backend, findings summary, from the `history` event stream. Cheap; it's also the marketing artifact.
7. **Response diffing / dedupe plugin** — "unchanged since call #41" or JSON-patch deltas for repeated identical calls, keyed by the cassette's canonical hash. ~150 lines, real savings on list-heavy agent loops.
8. **The consolidation refactors from analysis §1.2** — shared MCP-server bootstrap + `TargetSpawnOptions` + unified tool type (a-half-day each, do them opportunistically as the above touch those files) and the `ReplSession` de-singletonization when the REPL is next touched seriously.

### Tier 3 — Strategic Bets *(sequenced, each with a kill-switch)*

1. **`run-mcp eval`** — scripted/replayed agent loops against a target server; grades tool-selection accuracy, latency, schema compliance, `outputSchema` conformance; JSON verdicts for CI. *Also the instrument that settles the BM25-vs-embeddings question with your own data (§2).* Kill-switch: if nobody runs it in 3 months, stop investing.
2. **Serve the proxy over Streamable HTTP** — `run-mcp proxy --listen :8080` (stateless-capable, SDK server transport). Changes the deployment unit from CLI to sidecar/team service; prerequisite for any hosted story. Natural companion: `--stats` becomes a `/metrics`-ish surface.
3. **Skills bridge** — serve local `SKILL.md` directories through the DCL surface (frontmatter in the summary pool; body on activation; scripts/references as gated tools). Positions run-mcp across both packaging standards; zero heavy deps.
4. **Task-semantics shim + era bridge (design now, build when spec settles)** — wrap slow sync backends in MCP Tasks for task-capable clients; translate between protocol eras at the boundary. Both are man-in-the-middle exclusives; both depend on the Tier-2 §1 codec seam existing. Don't weave tasks through the core — the 2026 era reworks them into an extension.
5. **Code Mode (`run_script`), opt-in, sandboxed** — the compression staircase's terminal step: one script composes N namespaced tools (`tools.github.create_issue(...)`), collapsing round trips and fixing the token ceiling regardless of catalog size. Keep the deferred/opt-in posture on the sandbox dependency (your locked decision — still right); the *design* work (tool-identity → generated-API mapping) costs nothing now and constrains the multiplexer's naming (already compatible: `server__tool` ↔ `tools.server.tool`).
6. **Schema-aware result pruning** — `outputSchema`-guided (or shape-learned) field dropping with per-tool allowlists. The plugin nobody has; pairs with spill-to-disk (prune the inline head, keep everything in the handle).
7. **Embeddings (Stage C) — only on evidence** from Bet #1's retrieval benchmark. The module seam (`ranking.ts`) already permits a drop-in; resist building it on the research's say-so.
8. **The flight-recorder trajectory** — always-on bounded session recording + `run-mcp replay <session>` time-travel debugging + exportable repro bundles (cassette + config + transcript). Every Tier-1/2 item above (cassette batching, stats, audit, snapshot diff) doubles as a component of this. Treat it as the north star that sequences investments, not a single project to start.

### Tier 4 — New Projects / Spin-offs *(worthy ideas that would be scope creep here)*

1. **Enterprise MCP gateway** (OAuth JWT validation, RBAC, multi-tenant policy, central fleet management). The research confirms the market exists — and that Kong/Red Hat/TrueFoundry already occupy it. A spin-off would look like: run-mcp's passthrough core + policy engine extracted into a horizontally-scalable HTTP service with an identity-provider integration and an admin plane. **Recommendation: don't** — different buyer, different ops burden, incumbent competition; revisit only if a paying design partner shows up. The *local single-user* policy engine stays in run-mcp.
2. **Hosted agent-observability service** (the flight recorder as SaaS: team-shared session archives, diffing, alerting on findings). Plausible company; entirely different product surface (storage, auth, UI, privacy). The right split, if it ever happens: run-mcp stays the open-source recorder/emitter; the service consumes its artifacts. Nothing to do today except keep artifact formats (cassette, audit JSONL, session stats) versioned and documented — they'd be the interchange format.
3. **`@run-mcp/middleware` library** (`wrapClient(client, plugins)`) — per your locked decision, gated on a real external consumer **and** now also on the SDK v2 API settling (avoid building the library twice). Sketch when triggered: plugins package (types + built-ins, no CLI deps), a `wrapClient` adapter over the SDK `Client`, semver + a compat table against SDK majors. The prep work (freeze/version `InterceptorPlugin`) is already in Tier 1–2.
4. **Registry-aware discovery** (replacing the 15-path config scanner treadmill with MCP-registry lookups) — belongs upstream or as a plugin once the registry ecosystem stabilizes; not worth owning as core scope yet.

### Explicitly deprioritized
- **`--compress-aggressive` extensions / more whitespace tricks** — marginal by the research's own best numbers; structural compression (Tier 2 §2, §7; Tier 3 §6) dominates it.
- **TOON-style output re-encoding** — same reasoning; revisit only with eval-harness evidence.
- **MOQT transport** — interesting IETF draft, no ecosystem pull yet; track, don't build.
- **`search_all_local_mcp_servers` expansion** — parallelize it (Tier 1 §10 adjacent) but don't grow it; the multiplexer + `find_tools` is the strategic version of this feature.

## 4. Decisions locked in the Phase-4 interview (July 2026)

1. **Proxy bar = real workloads, soon.** The full-fidelity passthrough substrate is Tier-2 #1, ahead of further compression/DX features.
2. **Deployment shape = local sidecar per client (stdio).** Zero-config local stays the identity. No frontend HTTP transport or frontend auth in Tier 2; `--listen` remains a Tier-3 bet. The owned config file survives (the multiplexer needs it) but is per-client, not per-service.
3. **Flight recorder = directionally yes, don't reorganize.** It remains the trajectory that informs taste, not a driver of current sequencing or artifact-format work. Revisit after the passthrough substrate lands.
4. **OAuth for remote backends = Tier-2 core gap.** Wire the SDK `authProvider` through `TargetManager` (token storage, browser flow, refresh) after passthrough; it unblocks all modes at once.
5. **Unify B1/B2 proxy surfaces now.** Break the single-backend surface while it's days old, as part of the passthrough rebuild, so the surface changes exactly once. Single backend = multiplexer of one, prefixing off, `find_tools` available everywhere.
6. **REPL = maintenance mode.** No structural investment; keep it working. One exception: extract the sampling/elicitation approval logic into a testable module — that's protocol correctness, not UI polish. The broader `ReplSession` refactor is dropped from the schedule (apply only if forced).

Net effect on Tier 2 ordering: **passthrough substrate (with B1/B2 unification folded in) → result spill-to-disk → OAuth → config file → stats → response diffing**, with the §1.2 consolidation refactors done opportunistically inside the passthrough work.

## 5. The one-paragraph version

Fix the leaks and the proxy's four credibility gaps now (cache, sampling forwarding, reconnect, cassette I/O). Then spend the next cycle making "transparent" literally true — passthrough substrate with a version seam, spill-to-disk results, one unified surface, OAuth, an owned config file — because that's what fuses the dev-tool and production-proxy identities into one product. Place strategic bets in the order that generates evidence (eval harness first — it also settles the embeddings question), keep Code Mode and the library deferred exactly as you decided, adopt the skills bridge from the research, and treat the flight recorder as the north star. Decline the enterprise gateway unless a design partner pays for it.
